DIAGNOSTIC ANCHOR (skip-gated) -- bug C: SYMMETRIC arrival-time RoW on an uncontrolled
(right_before_left) junction. Distinct from scenario 26 (asymmetric 2-vehicle right-before-left,
already green): here a directed CONFLICT CYCLE with no clear priority winner.

SETUP: one single-lane right_before_left crossroads (node C), four STRAIGHT-through vehicles, one
per approach (ns: NC->CS, sn: SC->CN, ew: EC->CW, we: WC->CE), ALL departing t=0 from 100 m out at
rest -- perfectly symmetric, arriving together. sigma=0, Euler, actionStepLength=1, teleport off,
seed 42.

SUMO (golden): flows all four. The N-S axis (ns + sn, which are parallel and do NOT conflict) crosses
FIRST -- enterJunction t=17, exitJunction t=20; then the E-W axis (ew + we) t=19..24. 0 stuck, 0
teleports, all cleared.

ENGINE (baseline, this commit): DEADLOCKS -- all four drive ONTO the junction (enterJunction
t=17..19) and stop mid-junction, each blocking the next around the ring; none exits. 4/4 permanently
stuck.

ROOT CAUSE. The four straight movements form a directed right-before-left 4-cycle (each yields to the
movement on its right): NC->CS (link 1) yields to WC->CE (13); WC->CE (13) yields to SC->CN (9);
SC->CN (9) yields to EC->CW (5); EC->CW (5) yields to NC->CS (1). No movement has priority over its
own foe, and all arrive together, so the willPass pre-pass resolves WillPass=false for ALL four
(each is braking-to-yield). In the real PlanMovements pass the crossing gate's
`foeYieldsThisStep = !foe.WillPass` is then TRUE for every ego (its foe "won't pass"), so ALL four
proceed onto the junction simultaneously -> mutual mid-junction gridlock.

HOW SUMO BREAKS IT (verified in the vendored source, MSVehicle::planMoveInternal
MSVehicle.cpp:2818-2839): an EXPLICIT right-before-left deadlock detector. For a link in state
LINKSTATE_EQUAL (netconvert emits connection state "=" for the straight/left movements at a
right_before_left junction) with `myWaitingTime > 0` (ego already halted at the stop line), SUMO walks
the blocker chain `MSLink::getFirstApproachingFoe` (MSLink.cpp:1080 -- the closest willPass foe on each
foeLink) up to 100 hops; if the chain WRAPS BACK to ego's own link (`blocker.second == *link`) it has
found a circular-yielding deadlock, and breaks it RANDOMLY: `if (RandHelper::rand(getRNG()) <
threshold) setRequest = false`, with `threshold = 0.25` for a STRAIGHT movement and `0.75` for a turn.
That is: a deadlocked vehicle rolls its RNG each step and, with per-step probability 0.25 (straight),
ABORTS its junction request -- yields, letting the cycle unwind. Over a few steps the rolls
desynchronise the four and one axis goes, then the other (here N-S at t=17, E-W at t=19).

CONSEQUENCE FOR PARITY: this is a RANDOM tie-break keyed on the vehicle RNG. The engine's determinism
policy (CLAUDE.md / C1) is "distribution SHAPE ported faithfully, RNG STREAM is ours" (VehicleRng /
SplitMix64, never SUMO's RandHelper/MT19937), so which vehicle backs off on which step will NOT match
SUMO. Therefore EXACT @1e-3 FCD parity is NOT achievable for this scenario -- only STUCK-COUNT /
statistical parity (the deadlock breaks and all four drain, matching SUMO's 0 stuck, but the exact
per-step trajectory differs). golden.fcd.xml is kept as the "SUMO flows, N-S-then-E-W" reference, NOT
an @1e-3 target; RungSymRblStraightDiagTests asserts stuck==0 (the _diag stuck-count convention), not
FCD parity.

(straight) / 0.75 (turn) and, on success, force WillPass=false (abort request) so it yields and the
cycle unwinds. HIGH regression risk to the ~11 committed crossing scenarios (08/11/26/27/29/31/32/38/
